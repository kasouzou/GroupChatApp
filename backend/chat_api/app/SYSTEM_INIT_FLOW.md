# System Initialization 処理の流れ (初期化フロー)

このドキュメントは、バックエンドサーバーが起動した際、自動的に行われるデータベースのセットアップやマイグレーションの処理フローを説明します。

---

## 1. サーバー起動時の自動処理 (Lifespan)
**主な役割**: `main.py` に記述された `lifespan` 関数は、APIサーバーが動き始める直前に、必要な準備（DBの準備など）をすべて整えます。

### 処理フロー
1.  **DBマイグレーションの実行** (`migration_runner.py`)
    *   `schema_migrations` テーブルを確認します。
    *   まだ実行されていない「新しいテーブル定義（SQL）」があれば、順に実行します。
    *   `app_user_sessions` (セッション管理) や `chat_group_invites` (招待管理) などのテーブルが、ここで自動的に作成・調整されます。
2.  **テーブル定義の同期** (`init_db.py`)
    *   ソースコード上の「モデル定義」と、実際のDBの「テーブル定義」に差分がないか確認し、足りないテーブルがあれば作成します。
3.  **デモデータの投入 (Seeding)** (`init_db.py`)
    *   設定ファイル (`settings.py`) で `enable_demo_seed=True` になっている場合のみ動きます。
    *   開発の便宜上、あらかじめ「家族グループ」「プロジェクトA」などのテスト用データをDBに自動投入します。
    *   **本番環境では必ず `False` に設定されるため、勝手にデータが入ることはありません。**

---

## 2. データベース接続の管理 (`database.py`)
**主な役割**: APIがリクエストを受けるたびに、効率的にデータベースに接続するための窓口を提供します。

### 処理フロー
1.  **コネクションプールの構築**
    *   サーバー起動時にDBへの接続をあらかじめいくつか作っておき、使い回すことで高速化します。
2.  **セッションの貸し出し (`get_db`)**
    *   APIリクエストが来ると、1つの「セッション（接続）」を貸し出します。
    *   処理が終わると（成功してもしなくても）、自動的にセッションを閉じ、コネクションプールに返却します。
3.  **エラーハンドリング**
    *   もし処理の途中でエラーが発生した場合、`db.commit()` (確定) が行われず、変更内容はすべてキャンセルされます。

---

## 3. グローバルエラーハンドリング (`error_handlers.py`)
**主な役割**: 全てのAPIリクエストで発生した想定外の不具合を、一貫した形式でフロントエンドに伝える仕組みです。

### 処理フロー
1.  **エラーの検知**
    *   どんなAPIの処理中でも、エラー（例外）が発生したら捕まえます。
2.  **request_id の生成**
    *   不具合調査のために、そのエラー固有のID（`request_id`）をランダムに発行します。
3.  **レスポンスの統一**
    *   不具合の詳細をユーザーには見せず、「INTERNAL_SERVER_ERROR」などの共通メッセージと共に `request_id` をJSONで返します。
    *   バックエンド側のログには、その `request_id` と共にエラーの詳細（スタックトレース）が記録されます。
