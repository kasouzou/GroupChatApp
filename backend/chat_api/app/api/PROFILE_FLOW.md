# Profile Router 処理の流れ (プロフィールフロー)

このドキュメントは、ユーザープロフィールの表示、更新、および画像アップロードの処理フローを説明します。

---

## 1. プロフィール情報の取得
**エンドポイント**: `GET /api/v1/users/{user_id}`  
**主な役割**: Profile画面を表示する際に、最新のユーザー情報を取得します。

### 処理フロー
1.  **認証チェック**
    *   リクエスト者が本人（`user_id` と一致）か確認します。
2.  **ユーザーの検索**
    *   `app_users` テーブルから、そのユーザーの情報を取得します。
    *   **見つからない場合**: `404 Not Found` を返します（初回登録が必要な可能性をフロント側に知らせます）。
3.  **レスポンス返却**
    *   ユーザーID、名前、アイコン画像URL、作成日、更新日を返します。

---

## 2. プロフィール情報の更新
**エンドポイント**: `PUT /api/v1/users/{user_id}`  
**主な役割**: Profile画面で名前やアイコンを変更し、「保存」ボタンを押したときに呼ばれます。

### 処理フロー
1.  **認証チェック**
    *   リクエスト者が本人か確認します。
2.  **既存ユーザーの確認**
    *   更新対象のユーザーがDBに存在するか確認します。
3.  **情報の更新 (Update)**
    *   新しい `display_name` と `photo_url` を反映します。
    *   `updated_at`: **サーバー側の現在の時刻（UTC）**で更新日時を書き換えます。これにより、全端末で一貫した「最終更新日時」が保証されます。
4.  **データベースコミット**
    *   変更を確定（保存）します。
5.  **レスポンス返却**
    *   更新後の最新ユーザー情報を返します。

---

## 3. プロフィール画像アップロード (URL発行)
**エンドポイント**: `POST /api/v1/uploads/profile-image`  
**主な役割**: プロフィール画像を変更する際、フロントエンドがサーバーに画像のURLを問い合わせるAPIです。

### 現状の実装（モック実装）
現在はフロントエンドの開発を優先するため、**ダミーのURL**を返す最小限の実装になっています。
1.  リクエストボディから `file_path` を取得。
2.  ファイル名部分だけを抜き出して、仮のCDNドメイン (`https://cdn.example.com/...`) を付与して返します。

> **将来の展望**: 実際に AWS S3 や Google Cloud Storage などのストレージサービスに画像をアップロードし、その「署名付きURL（一時的に書き込み可能なURL）」をフロントエンドに返し、フロントエンドがそのURLに対して直接画像をバイナリデータとして送信する仕組みにアップグレードされる予定です。
