# Auth Router 処理の流れ (認証フロー)

このドキュメントは、ユーザーのログイン（Google連携）とログアウトの処理フローを説明します。

---

## 1. Googleログイン・ユーザー登録
**エンドポイント**: `POST /api/v1/auth/google-login`  
**主な役割**: Googleログイン後にフロントエンドから送られてくる情報を元に、DBへのユーザー登録（または更新）と、APIアクセス用の「合言葉（アクセストークン）」を発行します。

### 処理フロー
1.  **フロントエンドからのデータ受信**
    *   Googleから取得した `id` (uid), `display_name`, `photo_url` を受け取ります。
2.  **ユーザーの保存 (Upsert)**
    *   DBにその `id` のユーザーがいるか確認します。
    *   **いない場合**: 新しくユーザーを作成（初回登録）。
    *   **いる場合**: 名前やアイコン画像が変更されていれば最新の情報で上書きします。
3.  **セッション（ログイン状態）の発行**
    *   以前の古いセッション（トークン）を一度削除します（1端末1ログインを想定）。
    *   新しい「アクセストークン（ランダムな文字列）」を生成し、有効期限（例: 7日間）と共にDBに保存します。
4.  **レスポンスの返却**
    *   ユーザー情報一式と、生成した `access_token` をフロントエンドに返します。

> **フロント側の動き**: 受け取った `access_token` をローカルに保存し、以降のすべてのAPIリクエストのヘッダー (`Authorization: Bearer <token>`) に含めて送信します。

---

## 2. ログアウト
**エンドポイント**: `POST /api/v1/auth/logout`

### 処理フロー
1.  **認証チェック**
    *   送られてきたトークンが有効か確認します。
2.  **セッション削除**
    *   DBからそのユーザーのアクセストークン情報を削除します。これにより、同じトークンを使ってもAPIにアクセスできなくなります。
3.  **完了通知**
    *   `{"success": true}` を返します。

---

## 共通：認証の仕組み (`deps.py` の役割)
ほとんどのAPI（チャット送信やプロフィール取得など）には `@Depends(require_auth_user)` という記述があります。これは以下の「関所」を通ることを意味します。

1.  リクエストヘッダーからトークンを取り出す。
2.  DBの `app_user_sessions` テーブルにそのトークンがあるか探す。
3.  有効期限が切れていないか確認する。
4.  すべてOKなら「このユーザーは誰か」という情報を後続の処理に渡す。
5.  NG（トークンがない、期限切れなど）なら、処理を実行せずに `401 Unauthorized` エラーを返す。
