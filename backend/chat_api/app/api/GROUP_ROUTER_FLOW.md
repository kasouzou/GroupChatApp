# Group Router 処理の流れドキュメント

このドキュメントは `group_router.py` に定義されている3つのエンドポイントの処理フローを説明します。

---

## 1. グループ作成エンドポイント

**エンドポイント**: `POST /api/v1/groups`  
**レスポンス**: `CreateGroupResponse`

### 目的
NewChatタブで新しいグループを作成します。

### 処理フロー

```
1. 認証情報のバリデーション
   └─ リクエスト者が実際の作成者と一致しているか確認
      └─ 不一致 → 403 Forbidden エラー

2. 作成者ユーザーの確認
   └─ 作成者ユーザーがDBに存在するか確認
      └─ 存在しない → 404 Not Found エラー

3. グループの作成
   └─ グループID生成 (形式: grp_<UUID12文字>)
   └─ グループオブジェクト作成
      ├─ グループID
      ├─ グループ名
      ├─ 作成者ユーザーID
      └─ 作成日時（UTC）
   └─ DBセッションに追加

4. メンバーの登録
   └─ メンバーセット作成（作成者 + 指定メンバー、重複排除）
   └─ 各メンバーについて:
      ├─ ユーザーが存在するか確認
      ├─ 存在しない → スキップ（未登録ユーザーは無視）
      └─ 存在する → ChatGroupMemberレコード作成・追加

5. データベースコミット
   └─ すべての変更を永続化

6. レスポンス返却
   └─ グループID
   └─ グループ名
```

### エラーハンドリング

| ステータスコード | エラーメッセージ | 原因 |
|---|---|---|
| 403 | creator_user_id does not match auth user | リクエスト者と作成者が異なる |
| 404 | creator user not found | 作成者ユーザーが存在しない |

---

## 2. 招待コード生成エンドポイント

**エンドポイント**: `POST /api/v1/group-invites`  
**レスポンス**: `CreateInviteResponse`

### 目的
AddMember画面で、グループに新しいメンバーを招待するための招待コードを生成します。

### 処理フロー

```
1. 認証情報のバリデーション
   └─ リクエスト者が実際のリクエスター（招待者）と一致しているか確認
      └─ 不一致 → 403 Forbidden エラー

2. グループの確認
   └─ 指定されたグループIDでグループが存在するか確認
      └─ 存在しない → 404 Not Found エラー

3. リクエスターユーザーの確認
   └─ リクエスターユーザーがDBに存在するか確認
      └─ 存在しない → 404 Not Found エラー

4. メンバーシップの確認
   └─ リクエスターが指定されたグループのメンバーか確認
      ├─ SQLクエリでChatGroupMemberを検索
      └─ メンバーでない → 403 Forbidden エラー

5. 招待コードの生成
   └─ 招待コード生成 (形式: INV-<UUID10文字>、大文字)
   └─ 有効期限計算 (現在時刻 + 指定分数)

6. 招待レコードの作成
   └─ ChatGroupInviteレコード作成
      ├─ グループID
      ├─ 招待コード
      ├─ 発行者（リクエスター）ユーザーID
      └─ 有効期限
   └─ DBセッションに追加

7. データベースコミット
   └─ すべての変更を永続化

8. 招待URLの構築
   └─ リクエストのベースURLを取得
   └─ 招待URL構築 (形式: <base_url>/invite/<code>)

9. レスポンス返却
   └─ グループID
   └─ 招待コード
   └─ 招待URL
   └─ 有効期限（ISO 8601形式）
```

### エラーハンドリング

| ステータスコード | エラーメッセージ | 原因 |
|---|---|---|
| 403 | requester_user_id does not match auth user | リクエスト者とリクエスターが異なる |
| 403 | requester is not a group member | リクエスターがグループメンバーでない |
| 404 | group not found | 指定されたグループが存在しない |
| 404 | requester user not found | リクエスターユーザーが存在しない |

## 3. グループ参加エンドポイント

**エンドポイント**: `POST /api/v1/group-invites/join`  
**レスポンス**: `JoinByInviteResponse`

### 目的
招待コードを使用してグループに参加します（AddMember画面で招待を受ける側）。

### 処理フロー

```
1. 認証情報のバリデーション
   └─ リクエスト者が実際のユーザーと一致しているか確認
      └─ 不一致 → 403 Forbidden エラー

2. 招待コードの検索
   └─ 指定された招待コードでChatGroupInviteを検索
      └─ 見つからない → 404 Not Found エラー

3. 招待コードの有効性確認
   └─ 現在時刻を取得（UTC）
   └─ 招待コードの有効期限切れチェック
      └─ 期限切れ → 410 Gone エラー
   └─ 招待コードの使用済みチェック
      └─ 既に使用済み → 409 Conflict エラー

4. グループの確認
   └─ 招待に紐付くグループが存在するか確認
      └─ 存在しない → 404 Not Found エラー

5. ユーザーの確認
   └─ 参加しようとするユーザーがDBに存在するか確認
      └─ 存在しない → 404 Not Found エラー

6. メンバーシップの確認
   └─ ユーザーが既にそのグループのメンバーか確認
      ├─ SQLクエリでChatGroupMemberを検索
      └─ 結果に応じて new_joined フラグを設定

7. グループメンバーシップの登録（初回参加のみ）
   └─ メンバーシップが存在しない場合
      ├─ ChatGroupMemberレコード作成・追加
      └─ joined フラグを True に設定

8. 招待レコードの更新（使用済みマーク）
   └─ 招待を使用したユーザーIDを記録
   └─ 招待が使用された日時を記録

9. データベースコミット
   └─ すべての変更を永続化

10. レスポンス返却
    └─ グループID
    └─ グループ名
    └─ 新規参加フラグ (joined: boolean)
```

### エラーハンドリング

| ステータスコード | エラーメッセージ | 原因 |
|---|---|---|
| 403 | user_id does not match auth user | リクエスト者とユーザーが異なる |
| 404 | invite code not found | 招待コードが存在しない |
| 404 | group not found | グループが存在しない |
| 404 | user not found | ユーザーが存在しない |
| 409 | invite code already consumed | 招待コードが既に使用済み |
| 410 | invite code expired | 招待コードが有効期限切れ |

## データモデルの関連性

```
AppUser（ユーザー）
  ├─ ChatGroup（グループ）
  │   ├─ creator_user_id → AppUser
  │   └─ ChatGroupMember（グループメンバーシップ）
  │       ├─ group_id → ChatGroup
  │       └─ user_id → AppUser
  │
  └─ ChatGroupInvite（招待）
      ├─ group_id → ChatGroup
      ├─ created_by_user_id → AppUser（招待発行者）
      └─ consumed_by_user_id → AppUser（招待使用者）
```

---

## セキュリティと検証ポイント

1. **認証**: すべてのエンドポイントで `require_auth_user` 依存性により認証を確認
2. **認可**: リクエスト者と操作対象ユーザーの一致を検証
3. **グループメンバーシップ**: 招待コード発行時にメンバーシップをチェック
4. **招待コードの有効性**:
   - 有効期限の確認
   - 使用済みフラグの確認
5. **データ存在確認**: すべての外部キー参照がDB上に存在することを確認

---

## フローの特徴

- **非同期処理**: すべてのDB操作が非同期（async/await）で実装
- **トランザクション管理**: `db.commit()` で全変更をアトミックに反映
- **UUID活用**: グループID、招待コードに一意性の高いIDを使用
- **UTC時間**: すべての時刻管理をUTCで統一
- **柔軟なメンバー登録**: 未登録ユーザーはスキップ（運用ポリシーで変更可能）
