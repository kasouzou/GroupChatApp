# Chat Router 処理の流れ (チャットフロー)

このドキュメントは、チャット一覧の取得、メッセージ履歴の表示、メッセージ送信の処理フローを説明します。

---

## 1. ユーザー所属グループ一覧の取得
**エンドポイント**: `GET /api/v1/users/{user_id}/groups`  
**主な役割**: ユーザーが参加しているすべてのチャットグループと、各グループの最新メッセージを表示するためのサマリー情報を取得します。

### 処理フロー
1.  **認証チェック**
    *   リクエスト者が `user_id` と一致しているか確認します。
2.  **所属グループの検索**
    *   `chat_group_members` テーブルから、そのユーザーがどのグループ (`group_id`) に所属しているかを取得します。
3.  **グループ情報の集約**
    *   見つかったグループごとに情報を集めます。
    *   `latest_message`: そのグループで最後に送られたメッセージを取得し、プレビュー（テキストの一部）を表示用に加工します。
    *   `member_count`: そのグループに何人参加しているかをカウントします。
4.  **並び替え (Sort)**
    *   **最新メッセージが新しい順**（昇順）に並べ替えます。
5.  **レスポンス返却**
    *   グループID、グループ名、最新メッセージ、未読数（現在は暫定0）、メンバー数を返します。

---

## 2. メッセージ履歴の取得
**エンドポイント**: `GET /api/v1/groups/{group_id}/messages`

### 処理フロー
1.  **メンバーシップ確認**
    *   リクエスト者がそのグループのメンバーか確認します。
2.  **履歴の抽出**
    *   `chat_messages_remote` テーブルから、そのグループに属するメッセージを**古い順（時系列）**に最大200件取得します。
3.  **データ加工**
    *   メッセージの内容（テキストか画像か）に応じてレスポンスの形を整えます。
4.  **レスポンス返却**
    *   メッセージのリストを返します。

---

## 3. メッセージの送信
**エンドポイント**: `POST /api/v1/messages`  
**主な役割**: フロントエンドが「送信ボタン」を押したときに呼ばれるAPIです。サーバー側でIDと時刻を「確定」させます。

### 処理フロー
1.  **バリデーション**
    *   送信者が本人か、グループのメンバーかを確認します。
2.  **サーバー確定値の発行**
    *   `server_id`: サーバー側で一意なID（UUID）を生成します。
    *   `server_sent_at_ms`: クライアント側の時計を信用せず、**サーバーの現在の時刻（UTC）**を送信時刻として採用します。
3.  **メッセージの保存**
    *   `chat_messages_remote` テーブルに新規レコードを作成します。
    *   `local_id`: フロントエンドが管理している一時的なIDを紐づけます（これにより、フロント側は「送信中」から「送信完了」へ状態を更新できます）。
4.  **レスポンス返却**
    *   確定した `server_id` と `server_sent_at_ms` を返します。

> **データの流れ**: フロント側で作成した `local_id` が、サーバー側で `server_id` に紐付いて保存されることで、同期が完了します。
